Antes de publicar — Checklist de la app

🔍
Antes de tocar nada de tiendas de apps, la app tiene que funcionar. No perfecta — funcional. Sigue esta lista para confirmarlo.

wallet-control

  • La app abre sin crash en un celular real
  • El onboarding (primera pantalla) funciona completo
  • Se puede agregar una tarjeta y registrar un gasto
  • El historial muestra los gastos del mes
  • Finando IA responde (aunque sea en modo demo)
  • El modo oscuro y claro funcionan
  • La pantalla "Ayuda" tiene contenido (no está vacía)
  • El ícono de la app es el definitivo (no el genérico de Expo)

ruta-segura

  • La app abre sin crash en un celular real
  • El botón SOS lleva a la pantalla de emergencia
  • Se puede seleccionar un estado de USA
  • Las 5 categorías del menú principal abren
  • El formulario "Mi Caso" se puede llenar y enviar
  • El número de WhatsApp es el correcto (no el placeholder)
  • El selector de idioma Español/English funciona
  • El ícono de la app es el definitivo
⚠️ Importante: Si algún punto falla, arréglalo antes de continuar. Publicar una app con bugs visibles resulta en malas reseñas desde el día 1 y es muy difícil recuperarse.

Fase 1

Crear tu cuenta de desarrollador

🪪
Para publicar en una tienda oficial necesitas una cuenta de desarrollador. Es un trámite de una sola vez. Primero Android — es más rápido y barato.

🤖 Google Play (Android)

  • Ve a: play.google.com/console
  • Inicia sesión con tu cuenta Google
  • Paga $25 USD — pago único para siempre
  • Llena tu perfil de desarrollador (nombre, email de contacto)
  • Acepta los términos
  • Listo en 24-48 horas

🍎 App Store (iOS)

  • Ve a: developer.apple.com/programs
  • Necesitas un Mac o acceso a uno
  • Cuesta $99 USD por año
  • El proceso de aprobación toma 1-2 días
  • Recomendado: empieza por Android primero
  • iOS agrégalo cuando ya vendas en Android
💡 Consejo: Usa el mismo correo de Google para todo — Google Play Console, Firebase (si lo usas), y tu cuenta de desarrollador. Así todo queda unificado.
  • Cuenta de Google Play creada y verificada
  • Perfil de desarrollador completo (nombre real o nombre del proyecto)
  • Tienes acceso al panel de Google Play Console

Fase 2

Configurar EAS Build y generar el APK

📦
¿Qué es EAS? Es el servicio de Expo para convertir tu código en una app real (un archivo .apk para Android o .ipa para iOS) sin necesidad de Android Studio ni Xcode. Lo hace en la nube.

Paso 1 — Instalar EAS CLI (una sola vez)

# Instalar la herramienta globalmente
npm install -g eas-cli

# Verificar que quedó instalado
eas --version

Paso 2 — Iniciar sesión con tu cuenta Expo

# Logearse (se abre una ventana en el navegador)
eas login

# Verificar que estés logueado
eas whoami
💡 Si no tienes cuenta Expo: Ve a expo.dev y créala gratis. El plan gratuito te da 30 builds por mes — más que suficiente.

Paso 3 — Configurar el proyecto (hacer una vez por proyecto)

# Entrar a la carpeta del proyecto primero
cd C:\Users\Lenovo Ideapad\Documents\GitHub\claude-workspace\APPS\wallet-control

# Configurar EAS (genera el archivo eas.json automáticamente)
eas build:configure

Este comando te va a preguntar cosas. Responde así:

PreguntaRespuesta
Which platforms would you like to configure?Android
What would you like your Android application id to be?com.walletcontrol.app (o el que ya tienes en app.json)

Paso 4 — Generar el APK de prueba

# Build para pruebas internas (más rápido, sin firmar)
eas build --platform android --profile preview

Esto tarda entre 5 y 15 minutos. Al terminar te da un link para descargar el .apk. Instálalo en tu celular Android para probarlo.

Paso 5 — Generar el APK de producción (para subir a Play Store)

# Build oficial para publicar (firmado con keystore)
eas build --platform android --profile production
✅ Repite esto para ruta-segura: Entra a la carpeta APPS/ruta-segura y ejecuta los mismos comandos desde el Paso 3.
  • eas-cli instalado
  • Logueado en Expo con tu cuenta
  • wallet-control: eas.json generado + APK de prueba descargado e instalado
  • ruta-segura: eas.json generado + APK de prueba descargado e instalado
  • Ambas apps abren bien desde el APK instalado

Fase 3

Seguridad antes de publicar

🔐
Si alguien agarra tu API key o contraseña de base de datos, puede gasarle plata a tu cuenta o borrar todos los datos de tus usuarios. Esta fase evita eso.
🚨 Regla de oro: Ninguna contraseña, API key, ni secret puede estar escrita directamente en el código. Si está en un archivo que sube a GitHub, es como publicarla en Twitter.

¿Qué es una variable de entorno?

Es una forma de guardar información secreta (como contraseñas o API keys) fuera del código. En lugar de escribir la key directamente, escribes una referencia a ella. La key real solo existe en tu computador o en los servidores de Expo — nunca en GitHub.

wallet-control — Seguridad específica

  • La API key de Anthropic está en .env.local como EXPO_PUBLIC_ANTHROPIC_API_KEY=sk-ant-...
  • El archivo .gitignore tiene la línea .env*
  • La API key nunca aparece escrita directamente en ningún archivo .ts o .tsx
  • Agregar la API key a EAS Secrets para que funcione en el build de producción:
# Agregar secret en EAS (reemplaza YOUR_KEY con tu key real)
eas secret:create --scope project --name EXPO_PUBLIC_ANTHROPIC_API_KEY --value "sk-ant-YOUR_KEY_HERE"

ruta-segura — Seguridad específica

  • Cuando agregues Supabase (Fase de backend), las credenciales van en .env.local
  • El número de WhatsApp real está en el código (esto es público, no es un secret)
  • .gitignore tiene la línea .env*

Verificación general para ambas apps

# Buscar si hay secrets en el código (corre esto en cada carpeta)
cd APPS\wallet-control
Select-String -Path "**\*.ts","**\*.tsx","**\*.js" -Pattern "sk-ant-|password=|secret=" -Recurse

Si ese comando devuelve resultados, hay secrets en el código y hay que moverlos a variables de entorno antes de publicar.

  • Ningún secret hardcodeado en el código
  • .gitignore correcto en ambos proyectos
  • API keys en EAS Secrets para producción
  • Probado que la app funciona después de aplicar los cambios

Fase 4

Pruebas finales en celular real

📱
El emulador miente. Siempre prueba en un celular real con el APK instalado. Lo que funciona en el emulador a veces falla en Android real — y viceversa.

Flujo completo — wallet-control

  • Abrir la app por primera vez (como usuario nuevo)
  • Completar el onboarding: poner nombre, elegir avatar
  • Agregar una tarjeta débito con saldo
  • Agregar una tarjeta crédito con límite
  • Registrar un gasto via Finando IA en lenguaje natural
  • Ver que el gasto aparece en el Dashboard
  • Registrar un ingreso
  • Ver el Historial del mes
  • Crear una meta de ahorro
  • Exportar el reporte PDF
  • Cerrar y reabrir la app — verificar que los datos siguen ahí

Flujo completo — ruta-segura

  • Abrir la app por primera vez
  • Cambiar el idioma a inglés y verificar que todo cambia
  • Volver a español y crear una cuenta
  • Seleccionar un estado de USA
  • Explorar las 5 categorías del menú
  • Tocar el botón SOS y verificar que lleva a la pantalla de emergencia
  • Verificar que el botón de llamada al 911 funciona (abre el marcador)
  • Reportar un caso en "Mi Caso"
  • Probar en modo avión — verificar que la app no crashea

Pruebas de dispositivo

  • Probar en pantalla pequeña (5") — ¿todo se ve bien?
  • Probar en pantalla grande (6.7") — ¿todo se ve bien?
  • Probar con el celular en horizontal — ¿la app no se rompe?
  • Probar con fuente grande del sistema — ¿los textos caben?
⚠️ Si encuentras un bug aquí, arréglalo antes de continuar. Es mucho más costoso (en tiempo y reputación) arreglarlo después de publicar.

Fase 5

Preparar la ficha en Google Play

🎨
La ficha es lo que la gente ve antes de descargar tu app. Es literalmente la vitrina. Una buena ficha hace que descarguen; una mala, que sigan de largo.

Lo que necesitas tener listo

Elemento wallet-control ruta-segura
Nombre (máx 30 chars) Wallet Control Ruta Segura
Descripción corta (máx 80 chars) "Controla tus finanzas con IA en español" "Recursos y apoyo para mujeres migrantes en EE.UU."
Categoría Finanzas Estilo de vida / Social
Clasificación de contenido Para todos Para todos
Idioma principal Español Español (+ inglés)

Screenshots (capturas de pantalla)

Google Play requiere mínimo 2 screenshots. Recomendado: 6-8. Deben mostrar las mejores pantallas de la app. Tamaño mínimo: 320px × 320px.

💳 wallet-control

  • Dashboard principal (Hero card)
  • Chat con Finando IA
  • Pantalla de tarjetas
  • Historial con gráfica
  • Modo oscuro vs. claro
  • Metas de ahorro

🎀 ruta-segura

  • Pantalla de bienvenida
  • Menú principal (5 categorías)
  • Pantalla SOS
  • Detalle de categoría Legal
  • Formulario Mi Caso
  • Onboarding en inglés

Ícono 512×512

Google Play pide un ícono de 512×512 píxeles en PNG. El que tienes en assets/icon.png probablemente sirve — solo verifica que sea exactamente 512×512.

Política de privacidad

Google Play la exige para todas las apps. Necesitas una URL pública donde esté publicada. La forma más fácil:

  • Crear un documento en Google Docs con la política
  • Publicarlo como "cualquiera con el link puede ver"
  • Copiar el link y pegarlo en Google Play Console
💡 Política de privacidad mínima: Debe decir qué datos recopilas, cómo los usas y cómo el usuario puede eliminarlos. Para wallet-control (datos locales): "Los datos se guardan solo en tu dispositivo y no se comparten con terceros." Para ruta-segura: incluir que el modo anónimo no guarda ningún dato.
  • Nombre y descripción corta listos
  • Descripción larga escrita (mínimo 3 párrafos)
  • Mínimo 4 screenshots capturados
  • Ícono 512×512 listo
  • Política de privacidad publicada y con URL

Fase 6

Subir la app a Google Play

🚀
Google Play tiene 3 etapas antes de llegar al público general. Empezar por "Prueba interna" es lo más seguro — te da acceso inmediato sin esperar revisión.

Las 3 etapas de publicación

Etapa Quién puede descargarla Revisión de Google Recomendado para
Prueba interna Hasta 100 personas (tú + amigos) No Probar que todo funciona
Prueba cerrada Lista de emails que defines Sí (1-3 días) Beta testers reales
Producción Todo el mundo Sí (3-7 días) Lanzamiento público

Proceso paso a paso

  • Generar el AAB de producción con EAS:
# Desde la carpeta del proyecto
eas build --platform android --profile production

# Cuando termine, descarga el archivo .aab del link que te da
  • Ir a play.google.com/console
  • Crear nueva app (botón azul arriba a la derecha)
  • Llenar: nombre, idioma, app o juego, gratis o de pago
  • En el menú izquierdo: "Versiones" → "Prueba interna" → "Crear nueva versión"
  • Subir el archivo .aab
  • Llenar la ficha (nombre, descripción, screenshots, ícono)
  • Completar el "Panel de configuración" (política de privacidad, clasificación de contenido)
  • Publicar en prueba interna
  • Invitar testers (Correos → Agregar correos → Copiar link de opt-in)
  • Probar con los testers 2-3 días
  • Si todo está bien → pasar a Producción
⚠️ La primera revisión de Google puede rechazar la app por razones menores (descripción muy corta, política de privacidad incompleta). Si pasa, solo corrige lo que piden y re-envía. Es normal.

Fase 7

Estrategia de precio y venta

💰
Publicar la app no es venderla. Necesitas una estrategia de precio y una forma de que la gente la encuentre.

Modelo recomendado por app

💳 wallet-control

  • Modelo: Freemium
  • Gratis: funciones básicas (3 tarjetas, historial 3 meses)
  • Premium $2.99/mes: ilimitado + exportar PDF + IA avanzada
  • Implementación: in-app purchase de Google Play
  • Alternativa: APK directo en Gumroad por $4.99 sin restricciones

🎀 ruta-segura

  • Modelo: Gratis (misión social)
  • Sin cobro a las usuarias
  • Monetización: donaciones de organizaciones aliadas
  • Alternativa: botón de donación (Ko-fi / PayPal)
  • Valor: visibilidad, impacto social, puerta a financiamiento

Canales de lanzamiento

Canal Qué hacer Para qué sirve
Product Hunt Publicar el día de lanzamiento Visibilidad masiva, votos de la comunidad tech
Reddit r/PersonalFinanceApps, r/androidapps Feedback real + primeros usuarios
Twitter/X Thread mostrando el "build in public" Comunidad de makers e indie hackers
TikTok/Instagram Video demo de 60 segundos Alcance orgánico masivo
Comunidades latinas Grupos de Facebook, Telegram Usuarios objetivo directos (especialmente ruta-segura)

Opción alternativa sin App Store — Gumroad

Si no quieres esperar la revisión de Google Play o quieres vender antes, puedes subir el APK directamente a Gumroad y que la gente lo instale manualmente. Es legal y funciona.

  • Crear cuenta en gumroad.com
  • Subir el .apk como producto digital
  • Poner precio (o 0 para descarga gratuita)
  • Compartir el link

Fase 8

Después del lanzamiento

📊
El lanzamiento no es el final — es el inicio. Aquí está lo que tienes que hacer la primera semana y las semanas siguientes.

Primera semana

  • Revisar las reseñas todos los días y responder las negativas
  • Ver si hay crashs en Google Play Console → "Android vitals" → "Crashs"
  • Compartir el link a mínimo 10 personas conocidas y pedirles una reseña
  • Hacer el post de lanzamiento en Product Hunt

Cómo publicar una actualización

Cada vez que corrijas un bug o agregues algo nuevo, necesitas subir una nueva versión. El proceso es:

# 1. Subir el número de versión en app.json
# "version": "1.0.1" (cambia este número)
# "versionCode": 2 (siempre sube de 1 en 1)

# 2. Generar el nuevo build
eas build --platform android --profile production

# 3. Subir el nuevo .aab a Google Play Console → Producción → Nueva versión

Métricas mínimas a revisar cada semana

Métrica Dónde verla Qué significa
Instalaciones Play Console → Estadísticas Cuánta gente descargó
Retención Play Console → Android vitals Cuánta gente la sigue usando después de 7 días
Crashs Play Console → Android vitals Errores que la app tiene en celulares reales
Calificación Play Console → Reseñas Estrellas promedio y comentarios
✅ Meta realista para el primer mes: 50 instalaciones, calificación de 4.0+, y al menos 1 usuario que te dé feedback con su nombre. Eso es un lanzamiento exitoso.
